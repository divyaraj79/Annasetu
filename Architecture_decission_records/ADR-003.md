# ADR-003 — Business Rules

**Status:** Accepted

**Date:** July 2026

---

# Purpose

This document records the business rules currently enforced by the AnnaSetu backend.

Business rules belong exclusively inside the Service Layer.

Models define persistence.

Schemas define API contracts.

Services enforce domain behavior.

This document represents the business rules implemented for the backend foundation.

Authentication, authorization, validation, and lifecycle rules are included.

---

# Business Rule Philosophy

AnnaSetu follows a strict separation of responsibilities.

* Models define the database structure.
* Schemas validate requests and responses.
* Services enforce business rules.
* Routers coordinate HTTP requests and responses.

Business rules should never be implemented inside Models, Schemas, or Routers.

---

# User Rules

Current rules:

* Email must be unique.
* Phone number must be unique.
* User role cannot be changed implicitly.
* User must exist before creating related Restaurant or NGO profiles.
* Users are created exclusively through the RegistrationService.
* Deleted users cannot authenticate.
* Inactive users cannot authenticate.
* Users may modify only their own profile unless they are administrators.

Future rules:

* Ownership validation.
* Role-based authorization.

---

# Restaurant Rules

Current rules:

* User must exist.
* User must be active.
* User role must be RESTAURANT.
* Only one Restaurant profile may exist per User.
* Verification status defaults to `PENDING`.
* Pending restaurant registrations may be approved only by an administrator.
* Pending restaurant registrations may be rejected only by an administrator.
* Rejected pending registrations are permanently removed together with the associated User account.
* Deleted restaurants cannot be modified.
* Only the restaurant owner or an administrator may modify the restaurant.

Future rules:

* Ownership validation.
* Automatic geocoding.

---

* Pending NGO registrations may be rejected only by an administrator.
* Rejected pending registrations are permanently removed together with the associated User account.

# Donation Rules

Current rules:

* Restaurant must exist.
* Restaurant must not be deleted.
* Restaurant must be approved.
* Quantity must be greater than zero.
* Expiry time must be after cooked time.
* Maximum food lifetime is 10 hours.
* Completed donations cannot be modified.
* Only the donation owner or an administrator may modify a donation.
* Deleted donations cannot be modified.
* Cancelled donations cannot be modified.
* Expired donations cannot be modified.

Future rules:

* Automatic geocoding.
* Automation workflow trigger.

---

# DonationItem Rules

Current rules:

* Parent Donation must exist.
* Parent Donation must not be deleted.
* Parent Donation must not be completed.
* Parent Donation must not be cancelled.
* Parent Donation must not be expired.
* Food name is trimmed before validation.
* Food name cannot be empty.
* Quantity must be greater than zero.
* Duplicate food names within the same Donation are not allowed (case-insensitive comparison).
* Only the parent donation owner or an administrator may modify donation items.

Future rules:

* Bulk creation during AI automation.

---

# Need Rules

Current rules:

* NGO must exist.
* NGO must not be deleted.
* NGO must be approved.
* Quantity must be greater than zero.
* Completed Needs cannot be modified.
* Only the owning NGO or an administrator may modify a Need.

Future rules:

* Matching integration.

---

# Match Rules

Current rules:

* Donation must exist.
* Donation must not be completed.
* Donation must not be cancelled.
* Donation must not be expired.
* NGO must exist.
* NGO must not be deleted.
* NGO must be approved.
* Duplicate Match records for the same Donation–NGO pair are not allowed.
* Match status is controlled by workflow logic.
* Generic CRUD endpoints cannot arbitrarily change Match status.
* Completed matches cannot be modified.
* Distance is maintained as part of each Match.

Future rules:

* Score calculation.
* Candidate ranking.
* Winner selection.
* NGO confirmation workflow.
* Automatic reassignment.
* Matching orchestration.

# Validation Strategy

Validation is centralized inside the Service Layer.

Current validation categories include:

* Entity existence
* Relationship validation
* Verification validation
* Duplicate prevention
* Lifecycle validation
* Quantity validation
* Time validation

Validation is divided into two categories.

Schema validation

* Type validation
* String normalization
* Email validation
* Phone validation
* Length validation

Service validation

* Business rules
* Entity relationships
* Verification
* Lifecycle
* Duplicate prevention

# Orchestration Philosophy

CRUD Services enforce rules for individual entities.

Cross-entity orchestration services coordinate multiple CRUD Services.

Current orchestration services include:

* RegistrationService

Future orchestration services include:

* MatchingService
* AutomationService
* NotificationService

These services coordinate existing CRUD Services rather than duplicating business logic.

These services should orchestrate existing CRUD Services rather than duplicating business logic.

# Administrative Registration Workflow

Restaurant and NGO registration follows an administrative approval process.

Registration

↓

Verification Status = `PENDING`

↓

Administrator

↓

Approve → Verification Status = `APPROVED`

or

Reject → Permanently remove the pending organization and its associated User account.

This workflow applies only before an organization becomes an active business entity.

---

# Decision

Every business rule introduced into AnnaSetu must be implemented in the appropriate Service class.

Cross-entity workflows should be implemented as orchestration services that coordinate existing CRUD Services instead of bypassing them.
