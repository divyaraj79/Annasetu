# ADR-004 — API Design & Backend Engineering Standards

Status: Accepted

Date: July 2026

# Purpose

This document defines the engineering conventions used throughout the AnnaSetu backend.

Unlike previous ADRs, which describe the overall architecture and database design, this ADR focuses on implementation standards that every API endpoint and backend module should follow.

The goal is to keep the codebase consistent, maintainable, and predictable as new features are added.

# REST API Philosophy

AnnaSetu follows REST principles.

Resources are exposed through nouns rather than verbs.

Examples:

/users
/restaurants
/ngos
/donations
/needs
/matches

Administrative operations are exposed through dedicated administrative endpoints rather than modifying normal CRUD routes.

Administrative workflows are exposed through orchestration endpoints rather than being embedded into CRUD services.

Current examples include:

- Registration approval
- Registration rejection
- Pending registration review

# Layer Responsibilities

Every request follows the same lifecycle.

Client

↓

Router

↓

Schema Validation

↓

Service

↓

Model

↓

Database

↓

Response Schema

↓

Client

Responsibilities remain strictly separated.

Routers coordinate requests.

Schemas validate data.

Services enforce business rules.

Models represent persistence.

# Authentication

Protected endpoints require JWT authentication.

Authentication is performed through:

get_current_user()

The authentication layer is responsible for:

verifying JWTs
loading the authenticated user
rejecting deleted users
rejecting inactive users

Authentication never performs authorization.

# Authorization

AnnaSetu uses two authorization strategies.

Role-Based Authorization

Administrative endpoints use:

require_role(...)

Examples:

- Registration approval
- Registration rejection
- Pending registration review
- Administrative user management

# Ownership Authorization

Business entities follow an Owner-or-Admin policy.

Owners may modify only resources they own.

Administrators bypass ownership restrictions.

Ownership validation remains explicit inside routers.

# Validation Strategy

Validation is divided into two layers.

Schema Validation

Schemas validate:

data types
required fields
email
phone
whitespace trimming
length
formatting

Schemas never implement business rules.

# Service Validation

Services validate:

entity existence
verification status
lifecycle
duplicate prevention
relationships
business rules

Business rules never belong inside schemas.

# Transaction Strategy

Entity services never own transactions.

Entity services may call:

flush()

They never call:

commit()

rollback()

Transactions belong to the highest orchestration layer.

Currently:

- CRUD Routers
- RegistrationService

Future:

- AutomationService
- MatchingService
- NotificationService

RegistrationService owns transactions for multi-entity registration and administrative approval workflows, while CRUD routers own transactions for single-entity operations.

# Exception Handling

Business logic raises:

ValueError

Routers:

rollback transactions
re-raise exceptions

Global exception handlers produce consistent HTTP responses.

# Database Access

Database interaction always occurs through SQLAlchemy ORM.

Raw SQL should be avoided unless performance requirements justify it.

Application validation complements database constraints.

Neither layer replaces the other.

# Soft Delete Policy

Business entities are logically deleted using the `is_deleted` flag.

Deleted records:

- remain in the database
- preserve history
- preserve analytics
- preserve relationships
- are excluded from normal business operations

Soft deletion cascades through dependent business entities in the Service Layer rather than relying on database cascade behavior.

Exception:

Pending Restaurant and NGO registrations rejected during the administrative approval process are permanently removed together with their associated User account.

This exception exists because the registration never entered the normal business lifecycle and therefore does not require historical preservation.

# Naming Conventions

The project follows consistent naming.

Models

User
Restaurant
Donation

Schemas

UserCreate
UserUpdate
UserResponse

Services

UserService
DonationService

Routers

user_router.py
donation_router.py

Dependencies

get_current_user()

require_role()

Consistency is preferred over abbreviation.

# Internal System Access

Internal orchestration components (RegistrationService, future LangGraph agents, schedulers, notification workers, and matching workflows) are considered system components.

They invoke backend services directly rather than making HTTP requests to routers.

This ensures that:

business rules remain centralized
HTTP overhead is avoided
orchestration remains reusable

# Error Response Strategy

Business validation errors return:

400 Bad Request

Authentication failures return:

401 Unauthorized

Authorization failures return:

403 Forbidden

Missing resources return:

404 Not Found

Unexpected server failures return:

500 Internal Server Error

All responses follow a consistent JSON structure.

# Future Evolution

Future features should extend the existing architecture rather than replacing it.

New modules should:

- reuse existing CRUD Services
- preserve transaction ownership
- follow established validation patterns
- maintain separation of concerns
- implement cross-entity workflows through orchestration services rather than embedding business logic inside routers or models

# Decision

All future backend development should adhere to these engineering standards.

Any deviation should be documented through a new ADR before implementation.
