# ADR-001 — Overall System Architecture

**Status:** Accepted

**Date:** July 2026

---

# Purpose

This document describes the overall backend architecture adopted by AnnaSetu.

It defines how different layers of the application interact and establishes clear responsibilities for every component.

This architecture is intended to remain stable throughout the MVP and serves as the foundation for Authentication, AI Automation, and future integrations.

---

# High-Level Architecture

The backend follows a layered architecture.

Client

↓

FastAPI Routers

↓

Service Layer

↓

SQLAlchemy Models

↓

Database

Business requests always flow through this pipeline.

Infrastructure concerns such as dependency injection, database session management, and FastAPI request handling are intentionally kept outside the business architecture and do not alter the layered design.

---

# Layer Responsibilities

## Router Layer

Responsibilities:

* Receive HTTP requests
* Validate request bodies using schemas
* Invoke service methods
* Return API responses

Routers should contain almost no business logic.

Their primary responsibility is coordinating HTTP communication.

---

## Schema Layer

Responsibilities:

* Request validation
* Response serialization
* Type safety
* API contracts

Schemas exist only for communication between clients and the backend.

They never contain business rules.

---

## Service Layer

The Service Layer is the heart of the backend.

Responsibilities include:

* Business rule validation
* Entity existence validation
* Relationship validation
* Verification checks
* Lifecycle validation
* CRUD customization
* Database interaction

Business logic is intentionally centralized here.

Future orchestration services (Authentication, RegistrationService, MatchingService, AutomationService, etc.) will coordinate existing services rather than replacing them.

---

## Model Layer

Models define the persistent data structure.

Responsibilities:

* Database schema
* Relationships
* Foreign keys
* Constraints
* ORM mapping

Models intentionally remain free of business logic.

---

## Database Layer

Responsible for:

* SQLAlchemy Engine
* Session Management
* Database Connection
* Base Model

The database stores application state but does not enforce business workflows.

# Business Logic Strategy

Business logic belongs inside Services.

Examples include:

* Entity validation
* Duplicate prevention
* Verification checks
* Lifecycle validation
* Quantity validation
* Time validation
* Relationship validation

Models remain simple data structures.

Routers remain thin HTTP endpoints.

Schemas remain API contracts.

This separation keeps business logic centralized and reusable.

---

# Transaction Ownership Strategy

AnnaSetu follows a single-owner transaction model.

Entity Services are responsible for:

- Business rule validation
- Entity creation
- Entity updates
- Entity deletion
- Calling `flush()` when persistence is required

Entity Services never call:

- `commit()`
- `rollback()`

Transaction ownership belongs to the highest-level caller.

Current transaction owners include:

- CRUD Routers
- RegistrationService

Future orchestration services will also own transactions, including:

- MatchingService
- AutomationService
- NotificationService

Entity services continue to own persistence (`flush()`) but never transaction completion.

This architecture allows multiple entity operations to participate in a single atomic transaction.

↓

# Authentication Architecture

Authentication uses JWT access tokens.

Every protected request:

Client

↓

JWT Token

↓

Authentication Dependency

↓

Current User

↓

Router

↓

Service

The authentication layer is responsible for:

JWT verification
loading the authenticated user
rejecting deleted users
rejecting inactive users

Authentication never performs authorization decisions.

---

# Authorization Strategy

Authorization follows two principles.

Role-Based Authorization

Administrative functionality uses:

require_role(...)

Current administrative responsibilities include:

- Restaurant approval
- Restaurant rejection
- NGO approval
- NGO rejection

Future responsibilities include:

- Dashboard
- Analytics
- Manual Match Management
- Monitoring

Business resources follow an Owner-or-Admin model.

Examples:

User
Restaurant
NGO
Donation
Donation Item
Need

Ownership checks remain explicit inside routers.

Automation services bypass routers and invoke services directly.

# Validation Strategy

Validation is split into two layers.

Schema Validation

Responsible for:

type validation
email validation
phone validation
whitespace trimming
length validation
Service Validation

Responsible for:

business rules
relationship validation
lifecycle validation
verification checks

This separation keeps API validation independent from business rules.

---

# Soft Delete Strategy

All business entities implement soft delete.

Deletion updates:

is_deleted = True

instead of physically removing records.

Soft-deleted entities:

- remain queryable for auditing
- preserve historical relationships
- preserve analytics
- cannot participate in business workflows

Dependent business entities are soft-deleted through Service-layer cascade operations.

Cascade behavior is implemented as business logic rather than database cascade in order to preserve transaction control and future workflow customization.

# Data Integrity Strategy

AnnaSetu relies on both:

Database constraints

and

Application validation.

Database constraints enforce:

Primary Keys
Foreign Keys
Unique Constraints
Composite Unique Constraints

Services continue enforcing business-specific rules.

# Decision

This layered architecture is the official backend architecture of AnnaSetu.

Future development should extend this architecture rather than replacing it.

Business logic must continue to reside inside Services, while higher-level orchestration (Authentication, RegistrationService, MatchingService, AutomationService, etc.) coordinates existing services without bypassing them.
